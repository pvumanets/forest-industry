**Связанные документы:** `mvp-scope.md`, `metrics-registry.md`, `data-map.md`, `glossary.md`, `product-framing.md`, `company-context.md`, `ui-copy-guidelines.md`, [`development-plan-v1.md`](../development%20docs/development-plan-v1.md)

## 1. Резюме для руководства

Компания продаёт строганый сухой брус в Челябинске; показатели разбросаны по Excel и кабинетам сервисов. **Grove Pulse** — закрытое веб-приложение с еженедельным ручным вводом и дашбордами: один согласованный слой цифр для владельцев и ответственных без обязательного BI-навыка. В v1 — ввод по ролям (владельцы, маркетолог, руководители точек), просмотр сводов и отчётов с графиками, периоды неделя / месяц / квартал, без выгрузок в файл и без импорта истории до запуска.

---

## 2. Цели и метрики успеха релиза

**Бизнес-цели v1**

- Снизить время согласования «какая цифра правильная» за счёт одного места хранения и определений из `glossary.md` / реестра.
- Дать владельцам обзор по сайту, точкам, картам, Ozon и возвратам в выбранном периоде с возможностью углубления в отчёты.

**Продуктовые метрики успеха** (количественные пороги — по согласованию с Евгением после запуска; ориентиры см. `product-framing.md` §10)

- Регулярность заходов владельцев и вводящих.
- Доля недель с сохранённым вводом по всем обязательным полям (в т.ч. со значением 0).

## 3. Пользователи и сценарии

### 3.1 Роли и права

| Роль | Описание | Права |
|------|----------|--------|
| **Владелец** | Евгений, Павел (отдельные учётные записи) | Просмотр дашборда владельца и всех отчётов/графиков в рамках MVP; **нет** ввода метрик (если отдельно не заведена вторая учётка с ролью ввода — по умолчанию **не совмещаем роли в одной учётке**). |
| **Маркетолог** | Николай | Ввод блоков маркетинга, сайта, репутации, Ozon по `metrics-registry.md` и `data-map.md`; **просмотр дашбордов в v1 — как у владельца** (полные своды и отчёты MVP). Иное ограничение прав просмотра — возможно в последующих версиях, в v1 не вводим. |
| **Руководитель точки** | По одной или нескольким точкам | Ввод **только** метрик офлайн-точки (`OFF-*`) для закреплённых точек. |

**Правило учётных записей v1:** одна учётная запись = **одна роль**. Совмещение «владелец + маркетолог» (и любые другие пары ролей) в одном логине **не поддерживается**.

**Один человек — две точки:** допускается, что у **одной** учётки роли «руководитель точки» закреплены **обе** офлайн-точки. В интерфейсе ввода обязателен **переключатель активной точки** (одна форма — одна выбранная точка за раз); сохранение ввода не смешивает данные двух точек.

### 3.2 Сценарии (кратко)

1. **Владелец** открывает приложение → попадает на **главный дашборд владельца** → выбирает период (неделя / месяц / квартал) → видит блоки сводки → нажимает блок → открывается **отчёт с графиками** по теме блока с выбором периода/дат в рамках экрана.
2. **Маркетолог** выбирает отчётную неделю из доступных (до трёх предыдущих пн–вс) → заполняет все обязательные поля своих блоков → сохраняет (допускается **0** там, где нет факта); при необходимости открывает те же дашборды, что и владелец, для проверки картины.
3. **Руководитель точки** выбирает неделю → при двух точках — **переключает точку** → вводит `OFF-*` → сохраняет (допускается **0**).

## 4. Объём (Scope)

**Входит в v1:** по `mvp-scope.md` с уточнениями из этого PRD:

- Название продукта в UI: **Grove Pulse**.
- Стартовый экран после входа для роли **владелец:** главный дашборд владельца (не экран ввода).
- Ввод: все первичные поля MVP по реестру для соответствующей роли **обязательны** для сохранения недели; **ноль — допустимое значение** (явный обход отсутствия данных без опциональных полей).
- **Нет** баннера / напоминания о неполной неделе в v1 (отложено; см. `mvp-scope.md` v0.4+).
- Главный дашборд владельца и drill-down отчёты — §5.3.

**Не входит:** как в `mvp-scope.md` §3 (брак при отгрузке после v1, экспорт файлов, e-mail/push, план-факт, CRM и т.д.).

## 5. Функциональные требования

### 5.1 Авторизация и доступ

- Вход по логину и паролю (механизм сессии/JWT — в [`dev-handoff-spec.md`](../development%20docs/dev-handoff-spec.md)).
- Учётные записи и роли создаются **только** сидингом / вручную администратором развёртывания; саморегистрации нет.
- Контроль доступа: владелец не видит формы ввода чужих ролей; руководитель точки не видит чужие точки; маркетолог не редактирует `OFF-*`.

### 5.2 Ввод данных (формы, недельный цикл)

- Выбор **отчётной недели** из **до трёх предыдущих** календарных недель пн–вс (`mvp-scope.md`).
- Все поля из `metrics-registry.md` для данной роли на выбранную неделю **обязательны**; типы и единицы — по реестру и `data-map.md`.
- **Числовые поля:** допускается значение **0** (и для денег — «ноль» в согласованной точности), если факта нет — чтобы завершить сохранение недели.
- Подсказки «откуда взять цифру» — по полям из `data-map.md` (краткий текст в UI; детализация в handoff).
- Репутация и недельный контур — по регламенту в реестре / `data-map.md` (пн, снимки с приложений карт).

**Переключатель точки** (при двух точках на одной учётке руководителя): перед вводом пользователь выбирает точку А или Б; все поля формы относятся к выбранной точке до переключения.

**Возвраты (v1):** вводим и храним **как есть** по реестру. Офлайн — **`OFF-RET-N` / `OFF-RET-SUM` всегда в привязке к офлайн-точке** (та же форма контекста точки, что выручка и заказы). Ozon — **`OZ-RET-*`** в блоке Ozon (виртуальная точка канала). В формулах продукта возвраты **не** уменьшают выручку. Как именно на стороне бизнеса сопоставлять возвраты с продажами и неделями — **вне детализации v1**; при необходимости уточнят после запуска, без смены ID полей.

### 5.3 Просмотр: главный дашборд владельца и отчёты

**Главный дашборд владельца**

- После входа роль **владелец** открывает этот экран.
- **Выбор периода агрегации:** неделя / месяц / квартал (логика агрегатов — `metrics-registry.md`, репутация — последний снимок в периоде).
- **Блоки на главной** (свод за выбранный период):
  1. **Сайт** — ключевые показатели блока сайта за период (посетители по каналам, итого, отказы, длительность — по реестру).
  2. **Точки** — свод по **офлайн-точкам**: полный набор `OFF-*` (**выручка, заказы, возвраты по каждой точке**, производные). Возвраты на дашборде **остаются в разрезе точки** (не «отвязанный» пул).
  3. **Карты** — два подблока: **2ГИС** и **Яндекс.Карты** (репутация: оценка, число отзывов, приросты по реестру).
  4. **Ozon** — полный набор по каналу: выручка, заказы, **возвраты**, при необходимости реклама, производные (`OZ-*`).
  5. **Возвраты** — **обязательный** отдельный блок: **свод по всей компании** за период (офлайн по всем точкам + Ozon: количества и суммы и при необходимости производные по реестру). **Не** отменяет привязку в данных к точкам/Ozon и не заменяет разрез в блоках «Точки» и «Ozon».

- **Сравнение с предыдущим периодом на главной (блоки п.1–5):** для **каждого** блока у **ключевых** KPI — значение за **предыдущий период того же типа** (предыдущая неделя / предыдущий месяц / предыдущий квартал) и **процент изменения**, где процент осмыслен; при **нуле в знаменателе** — правила **«—»**, **«нов.» / «рост с нуля»** и tooltip — в [`dev-handoff-spec.md`](../development%20docs/dev-handoff-spec.md) §6.2 и [`ui-copy-guidelines.md`](./ui-copy-guidelines.md) §4. Для **репутации** в месяце/квартале сравниваются агрегаты по правилу «последний снимок в периоде» (`metrics-registry.md` §3) между соседними периодами.

*Примечание:* детальная вёрстка, порядок блоков и состав «ключевых» цифр внутри каждого блока — в [`dev-handoff-spec.md`](../development%20docs/dev-handoff-spec.md); новые метрики сверх реестра не вводятся.

**Drill-down (отчёт с графиками)**

- Клик по **каждому** блоку главной открывает **отдельный экран отчёта** по тематике блока.
- На экране отчёта: **графики** динамики по доступной истории, **выбор диапазона дат / периода** в пределах реализованного UI, фильтры в объёме MVP (например, точка, площадка карт — по смыслу блока).
- Эталон качества графиков — десктоп; мобильный — упрощённо (`mvp-scope.md`).

### 5.4 Сравнение периодов

- **Год к году (YoY)** в v1 **не** входит.
- На **главной** дашборда владельца для **всех блоков** сводки (§5.3 п.1–5) обязательно отображение **предыдущего периода того же типа** и **процента изменения** для ключевых KPI — см. §5.3.
- На экранах **drill-down** и прочих графиках обязательство «предыдущий период + %» **не** распространяется на каждый виджет — по умолчанию выбранный период и подсказки по точке; детали — [`dev-handoff-spec.md`](../development%20docs/dev-handoff-spec.md).

### 5.5 Справочники и настройки

- **Две офлайн-точки** v1: коды **`NOVOGRAD`** (Точка на Новоградском) и **`SVERDLOV`** (Точка на Свердловском) — норматив в `glossary.md` (**Справочник точек v1**); каналы и прочее — по `glossary.md` и `mvp-scope.md`. Редактирование справочника точек через UI в v1 не обязательно.

### 5.6 Заложенное на будущее (без реализации в v1)

- Баннер / напоминание о неполной неделе в приложении.
- Индикатор полноты недели в UI (при необходимости вернуться к `product-framing.md` §6 п.5).
- Экспорт, брак при отгрузке, план-факт — см. `mvp-scope.md` и `product-framing.md`.

## 11. Критерии приёмки (уровень PRD)

1. Под учёткой **владельца** после входа открывается **главный дашборд** с выбором периода и блоками из §5.3 (включая отдельный блок **«Возвраты»** — свод по компании).
2. На главной для **всех блоков** п.1–5 §5.3 отображаются ключевые KPI за выбранный период и **предыдущий период + %** согласованно с §5.3–5.4 (где % осмыслен).
3. Под учёткой **маркетолога** доступны **те же** дашборды и отчёты просмотра, что у владельца (v1).
4. Клик по каждому блоку главной открывает отчёт с **графиками** и выбором периода/дат в рамках реализации.
5. **Одна роль — одна учётка**; сценарий руководителя с **двумя точками** и переключателем работает без перезаписи чужих данных.
6. Сохранение недели возможно только при заполнении **всех** обязательных полей роли; **0** принимается там, где тип числовой.
7. В v1 **нет** баннера о неполной неделе (согласованное отличие от раннего черновика `mvp-scope`).

Детальные Given/When/Then — в [`dev-handoff-spec.md`](../development%20docs/dev-handoff-spec.md).
