## 1. Определение MVP

**MVP (v1)** — продукт **Grove Pulse**: закрытое веб-приложение с авторизацией; **ручной ввод** всех метрик со статусом **MVP** из `metrics-registry.md` по **отчётным неделям** (пн–вс); **дашборды** (главный дашборд владельца с периодом неделя/месяц/квартал и drill-down отчётами с графиками — см. `prd.md`); роли — владельцы, маркетолог, руководители точек; **справочник из двух точек** (коды жёстко при запуске). **Валидация ввода:** все обязательные поля по роли должны быть заполнены; **0 допускается** при отсутствии факта (`prd.md`). **Баннер и индикаторы полноты недели в UI в v1 не делаем** — отложено. Данные **накапливаются с первой фактически заполненной недели** после запуска (**без импорта истории** из Excel). **Экспортов файлов и сценария «брак при отгрузке» в v1 нет** — см. §3.

## 2. Входит в MVP (Must have)

| № | Возможность | Кратко | Связь с метриками / экранами |
|---|-------------|--------|------------------------------|
| 1 | Авторизация | Вход в систему по учётным записям, созданным вручную (сидинг) | Все экраны за логином |
| 2 | Роли и доступ | Владельцы — обзор по компании; маркетолог — ввод своих блоков **и просмотр дашбордов как у владельца** (v1); руководитель точки — ввод **только по своей** точке. **Одна учётка = одна роль** (без совмещения). Один руководитель на **две** точки — **переключатель точки** во вводе (`prd.md`) | См. `product-framing.md` §4, `glossary.md`, `prd.md` |
| 3 | Справочник точек | **Две** точки с кодами **`NOVOGRAD`** / Точка на Новоградском и **`SVERDLOV`** / Точка на Свердловском (`glossary.md`); задаются **жёстко** при развёртывании/сиде (редактирование через код/миграции, не обязательно UI-админка в v1) | OFF-*, REP-* (разрез точка) |
| 4 | Выбор отчётной недели | Пользователь **явно выбирает** неделю ввода; UI предлагает **до трёх предыдущих** отчётных недель с подписью диапазона **пн–вс**. **Дедлайнов и блокировок** по календарю в системе **нет** | Недельный контур `glossary.md` |
| 5 | Ввод по офлайн-точкам | Выручка, заказы, возвраты (кол-во и сумма) за выбранную неделю | `metrics-registry.md` §1 |
| 6 | Ввод маркетинга и сайта | Расходы контекст / карты / Ozon-реклама; **посетители** сайта по каналам (Метрика); поведение сайта | §2, §2.0–2.1, `data-map.md` |
| 7 | Репутация | Снимки оценки и числа отзывов по точке×площадка; приросты считает система; **регламент ввода — пн** (см. `metrics-registry.md` §3, `data-map.md`) | §3 |
| 8 | Блок Ozon | Выручка, заказы, возвраты, реклама (одно поле `OZ-AD-SPEND` в блоке Ozon) | §4 |
| 9 | Производные и своды | Средние чеки, DER-*, WEB-TRF-TOT, DER-AD-TOTAL и т.д. по формулам реестра | §1.1, §2.2, §3.1, §4.1, §5 |
| 10 | Дашборды и периоды | Главный дашборд: выбор **неделя / месяц / квартал**; блоки сайт, точки, карты (2ГИС и Я.Карты раздельно), Ozon, **возвраты (отдельный свод по компании)**; на главной у ключевых KPI — **предыдущий период + %** во **всех** блоках (`prd.md`). **Клик** — отчёт с графиками. Агрегаты — `metrics-registry.md`; репутация в периоде — **последний снимок** | §3, §5, `prd.md`, `metrics-registry.md` §3 |
| 11 | Полнота ввода | Все поля MVP по роли **обязательны**; **0** допустим. **Баннер/индикатор полноты в UI — не в v1** | `prd.md` |
| 12 | Устройства | **Обязательно удобно с телефона:** ввод и просмотр результатов. **Графики:** эталон — **десктоп**; на мобильном допустим упрощённый вид, без требования «идеальных» графиков | NFR для `prd.md` / [`dev-handoff-spec.md`](../development%20docs/dev-handoff-spec.md) |
| 13 | Старт данных | Начало учёта **с текущей / первой выбранной** недели после запуска; **без** загрузки прошлых недель из файлов | Зависимости запуска §5 |

*Обязательный набор полей по неделе = все первичные MVP-метрики из реестра для соответствующей роли; ноль — допустимое значение; детали — `prd.md`.*

## 3. Не входит в MVP (Won’t have — v1)

| № | Возможность | Комментарий (когда рассмотреть) |
|---|-------------|----------------------------------|
| 1 | **Брак при отгрузке** | **Сразу после v1** — приоритетный следующий крупный блок; см. `product-framing.md` §7.2 |
| 2 | **Выгрузки** (CSV, Excel, отчёты в файл, печать) | **v2**, в связке с развитием после брака и прочими экспортами — в v1 **только просмотр в UI** |
| 3 | Напоминания по e-mail / push / баннер в приложении | Не делаем в v1 (`prd.md`); баннер полноты — отложен |
| 4 | Дедлайн закрытия недели в системе | Нет автоматической блокировки ввода по дате; регламент — вне продукта |
| 5 | Импорт исторических недель из Excel | Старт «с нуля» с момента запуска |
| 6 | Саморегистрация и UI-администрирование пользователей | Все пользователи и отображаемые имена — **ручной сидинг** ТПМ/владельца |
| 7 | План-факт, планы с разбивкой по неделям | v2+ |
| 8 | Воронка, CRM, call-tracking, сквозная аналитика | v3+ / после инфраструктуры |
| 9 | Сравнение периодов на дашборде | **Год к году (YoY)** в v1 **нет**. На **главном** дашборде — **предыдущий период + %** у ключевых KPI **во всех** блоках сводки (см. `prd.md`, `metrics-registry.md`). На drill-down — без обязательства на каждый виджет |
| 10 | Чистая выручка, нормативная привязка возвратов к дате продажи | Вне v1 по `metrics-registry.md` |

## 4. Архитектурные «крючки» без UI

| Закладка | Минимальное требование к v1 |
|----------|---------------------------|
| Выгрузки и отчёты в файл | **Не** входят в v1. При проектировании схемы БД **не принимать решений**, которые **навсегда** закроют возможность выборочного экспорта в v2 (достаточно нормализованного хранения фактов ввода и снимков) |
| Плановые значения (план-факт) | Не реализуем; при необходимости — отдельные таблицы только в v2 |
| Расширение справочника точек / каналов | Две точки и закрытый список каналов в v1; модель должна допускать добавление без ломки ключей |
| Аудит действий пользователей | Не обязателен в v1 (`agent_product.md` §7) |

## 5. Зависимости запуска

- **Учётные записи:** заведены вручную; пользователям заданы **имена** (отображение в UI по согласованию с ТПМ).
- **Справочник точек:** **`NOVOGRAD`**, **`SVERDLOV`** и отображаемые имена — **зафиксированы** до первого ввода (жёстко в сиде/конфиге; норматив `glossary.md` **Справочник точек v1**).
- **Справочник каналов:** как в `glossary.md` (в т.ч. Ozon отдельно; Авито по умолчанию не активен).
- **Регламент:** в продукте **нет** принудительного дедлайна; организационно можно ориентироваться на понедельник (`glossary.md`), но система только предлагает недели из **последних трёх** для выбора.
- **Данные:** первая сохранённая неделя = начало истории; прошлое не переносим.

## 6. Критерий «MVP готов»

Блок **важный** и может уточняться после первого полного прохода демо; ниже — **рабочий чеклист** для согласования с Евгением (владелец продукта) и ТПМ.

| # | Критерий | Проверка |
|---|----------|----------|
| A | **Инфра:** приложение доступно по согласованному URL (prod), **HTTPS** | Открытие в браузере, нет критичных предупреждений сертификата |
| B | **Сид:** 2 точки с кодами **`NOVOGRAD`** и **`SVERDLOV`**; пользователи всех ролей созданы; роли и привязка руководителей к точкам верны; сценарий **один руководитель — две точки** + переключатель | Матрица доступа: владелец видит свод; маркетолог — **те же** дашборды + свои формы; руководитель точки — только свои точки через переключатель |
| C | **Ввод:** для недели из **последних трёх** (пн–вс) сохраняются все обязательные поля MVP по роли; **0** допустим | Сценарий без обхода валидаций |
| D | **Дашборды:** главный экран владельца с периодом и блоками; drill-down с графиками; репутация в месяце/квартале — **последний снимок** | Сверка с ручным расчётом на тестовых данных |
| E | **Сравнение на главной:** для **всех** блоков сводки (сайт, точки, карты, Ozon, возвраты) у ключевых KPI — предыдущий период и % при наличии базы | Проверка на 2+ неделях (или периодах) истории |
| F | **Мобильный:** ввод и просмотр ключевых цифр **удобны** на типовом смартфоне; графики на десктопе — основной эталон качества | Проверка на 1–2 реальных устройствах |
| G | **Приёмка владельца:** Евгений подтверждает, что v1 закрывает согласованный управленческий радар на старте | Короткий протокол или письмо |

*Дополнительно можно ввести порог «N недель подряд с полным вводом» — **после** запуска, как продуктовую метрику, а не как блокер релиза.*
