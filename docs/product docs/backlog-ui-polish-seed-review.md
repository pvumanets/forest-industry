## 1. Продуктовый контекст

Сейчас функциональность v1 закрывает поток «ввод → сводка → отчёты», но **визуальный слой** и **ощущение от демо-данных** не дотягивают до уровня «показать владельцу без извинений». Это нормальный этап: сначала честный скелет, потом полировка и осмысленные сценарии в данных.

**Цель спринта (условно «день красоты и ясности»):**

1. Понять, **что именно** показывают синтетические данные и достаточно ли их для проверки гипотез UI/UX.
2. **Навести визуальный порядок** без смены продуктовой логики.
3. Зафиксировать **UI-гайд** так, чтобы и дизайн, и разработка ссылались на один артефакт (рядом с уже существующими правилами копирайта).

## 2. Эпик A — Синтетические данные: обзор и решения

Исходная формулировка: *«разобраться, как сейчас выглядят синтетические данные»*.

### A1. Карта сидов для демо (продукт / аналитика)


| ID       | Задача                                                                                                                                               | Критерий готовности                                                                     |
| -------- | ---------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------- |
| **A1.1** | Зафиксировать в одном месте **какие сущности** создаёт сид: пользователи, точки, недели, заполненность форм маркетинга/офлайна, дашбордные агрегаты. | Короткая таблица или раздел в этом файле + ссылка на `seed/data.py` и связанные модули. |
| **A1.2** | Пройти сценарии **по ролям** (owner / marketer / site_manager): что видно на дашборде и в отчётах, где «пусто» vs «заполнено».                       | Чеклист ролей с скриншотами или пунктами «ожидаю / факт».                               |
| **A1.3** | Оценить **правдоподобность** цифр для внутренних обсуждений (не как юридическая точность, а «не режет глаз владельцу»).                              | Список замечаний: OK / подправить в сиде / оставить как абстрактное демо.               |


**Комментарий к реализации (как вижу работу):**

- Код сидов уже структурирован по логинам и ролям — полезно **не раздувать** сид до «реалистичного бизнеса», а добавить **2–3 явных сценария**: например «полная неделя», «частично заполненная», «предыдущий период = 0» для проверки бейджей % на дашборде (см. `[ui-copy-guidelines.md](./ui-copy-guidelines.md)` §4).
- Если решите расширять данные — изменения в **одном** модуле/наборе фикстур + короткая заметка в README API «что в сиде для UI».

## 3. Эпик B — «Навести красоту» (визуальная полировка UI)

Исходная формулировка: *«пока всё выглядит не очень красиво»*.

### B1. Базовая визуальная система


| ID       | Задача                                                                                                                                          | Критерий готовности                                                                                          |
| -------- | ----------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------ |
| **B1.1** | Согласовать **палитру** (фон, поверхности, акцент, текст, границы, состояния ошибки/успеха) — пусть даже минимальная, без отдельного дизайнера. | CSS-переменные или Tailwind theme extension + 1 страница-пример в Storybook **или** раздел в UI-гайде с hex. |
| **B1.2** | **Типографика**: иерархия заголовков, подписи к графикам, плотность на формах ввода.                                                            | Единые классы/токены; нет «случайных» размеров шрифта на соседних экранах.                                   |
| **B1.3** | **Отступы и сетка**: единая сетка для карточек дашборда, форм, шапки.                                                                           | Повторяющиеся отступы (например кратность 4/8px) в ключевых layout-компонентах.                              |


### B2. Компоненты и состояния


| ID       | Задача                                                                           | Критерий готовности                                                            |
| -------- | -------------------------------------------------------------------------------- | ------------------------------------------------------------------------------ |
| **B2.1** | Кнопки, поля, карточки — **единый стиль** (hover, focus, disabled).              | Focus visible для клавиатуры; согласовано с тёмной темой если остаётся тёмной. |
| **B2.2** | **Пустые и загрузочные** состояния (дашборд, отчёты, формы) — не «пустой экран». | Тексты из гайда + скелетон или спиннер по единому правилу.                     |
| **B2.3** | **Графики (Recharts)**: легенда, оси, tooltip, контраст линий/областей.          | Читаемость на ноутбуке; не перегруженная легенда.                              |


### B3. Оболочка и навигация


| ID       | Задача                                                                                                 | Критерий готовности                                                                      |
| -------- | ------------------------------------------------------------------------------------------------------ | ---------------------------------------------------------------------------------------- |
| **B3.1** | Шапка / боковое меню / активный пункт — визуально ясно, где пользователь.                              | Соответствие пунктов `[dev-handoff-spec.md](../development%20docs/dev-handoff-spec.md)`. |
| **B3.2** | Адаптив **хотя бы** до «узкий ноутбук»: горизонтальный скролл таблиц осознанный, а не поломка вёрстки. | Smoke-проверка ширин 1280 / 1024.                                                        |


**Комментарий к реализации:**

- Делать **слоями**: сначала токены и layout, потом формы, потом графики — иначе придётся перекрашивать Recharts дважды.
- Не смешивать с **продуктовыми** изменениями (новые метрики, новые поля) в том же PR — проще ревью и откат.
- Имеет смысл завести **корневой layout-компонент** или расширить существующий, чтобы отступы страниц не копировались в каждом роуте.

## 4. Эпик C — UI-гайд (расширение артефакта)

Исходная формулировка: *«сделаем UI гайд»*.

Сейчас `[ui-copy-guidelines.md](./ui-copy-guidelines.md)` закрывает **тексты**. Визуал и паттерны компонентов логично оформить отдельно, чтобы не раздувать копирайт-файл.


| ID     | Задача                                                                                                                                                      | Критерий готовности                                                                           |
| ------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------- |
| **C1** | Создать `ui-visual-guidelines.md`: палитра, типографика, отступы, кнопки, поля, карточки, графики, состояния loading/empty/error. | **Выполнено** — см. [`ui-visual-guidelines.md`](./ui-visual-guidelines.md) v1.0; токены в коде: [`apps/web/src/index.css`](../../apps/web/src/index.css) (`:root`, `.dark`, `@theme inline`). |
| **C2** | Явная связка **копирайт ↔ визуал**: в начале обоих файлов взаимные ссылки.                                                                                  | **Сделано:** шапки `ui-copy-guidelines.md` и `ui-visual-guidelines.md` + §1.2 копирайта.                                            |
| **C3** | Раздел **«Проверка перед демо»** (короткий чеклист): контраст, ошибки форм, мобильность не обязательна, но зафиксировать что проверяем.                     | 10–15 пунктов, пригодно для ручного прогона перед показом владельцу.                          |


**Комментарий к реализации:**

- Документ в Markdown достаточен для v1; позже можно вынести **примеры в Storybook** — но это уже отдельная задача, если решите инвестировать время.
- В `[agent_product.md](./agent_product.md)` строка таблицы артефактов должна указывать на новый файл (обновление при мерже C1).

## 5. Предлагаемый порядок работ (завтра)

1. **A1.1 → A1.2** (30–60 мин): быстро понять картину данных без правок кода.
2. ~~**C1** черновик~~ — заменено на [`ui-visual-guidelines.md`](./ui-visual-guidelines.md) + тема Tailwind v4 / Tweakcn в `apps/web`.
3. **B1** + **B2.1** как основа визуала (дальнейшая полировка экранов под гайд).
4. **B2.2–B2.3**, **B3** по приоритету экранов (дашборд → отчёты → формы).
5. **A1.3** — решения, нужен ли рефактор сидов.

## 6. Что согласовать с владельцем продукта (если понадобится)

- Тёмная тема остаётся **основной** или целимся в светлую для «офисных» демо?  
- Допустимы ли **нейтральные** цвета для роста/падения KPI (как в копирайт-гайде) без «зелёный = хорошо»?  
- Нужны ли в сиде **имена точек/недель**, узнаваемые для Евгения/Павла, или достаточно абстрактных кодов?
