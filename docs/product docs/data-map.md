**Версия:** 0.6

## 1. Цель документа

Зафиксировать цепочку **метрика (ID) → источник правды as-is → кто вводит в систему → период**, без интеграций в MVP: все значения **ручной ввод** в веб-формы. Детали экранов и валидаций — в `prd.md` / [`dev-handoff-spec.md`](../development%20docs/dev-handoff-spec.md).

---

## 2. Сущность «отчётная неделя»

- **Календарная неделя пн–вс** (как в `glossary.md`). В интерфейсе пользователь выбирает неделю из **до трёх предыдущих** отчётных недель (`mvp-scope.md`); жёсткого дедлайна в системе нет.
- **Регламент команды:** ориентир — **понедельник**, ввод за **предыдущую** пн–вс (маркетолог, руководители точек; репутация — в том же проходе).
- **Ключ в модели данных:** однозначная привязка фактов ввода к паре «граница недели» (например, дата понедельника или ISO-год+неделя) — уточняется в [`architecture-draft.md`](../development%20docs/architecture-draft.md) / ТЗ.

## 3. Матрица: метрика × источник × исполнитель

Разрез **точка** для офлайн-метрик: **одна логика источника** на все точки; физически учёт может отличаться, **операционный регламент** согласовывает владелец/ТПМ вне продукта.

| Метрика (ID) | Источник правды сегодня | Кто вводит в систему | Периодичность | Комментарий |
|--------------|-------------------------|----------------------|---------------|-------------|
| OFF-REV, OFF-ORD | Учёт точки: **касса**, **Excel**, иные согласованные источники; полнота на старте **не гарантирована** | Руководитель **своей** точки | Раз в выбранную отчётную неделю | Целевой регламент сбора — вне v1 продукта; в систему — одна выручка и заказы за пн–вс |
| OFF-RET-N, OFF-RET-SUM | **Отдельный** файл/учёт возвратов точки (не смешивается с выручкой в продукте) | Руководитель точки | Раз в отчётную неделю | Не уменьшают OFF-REV; в продукте — **привязка к точке**; прочая бухполитика — вне детализации v1 |
| MKT-AD-CTX | Кабинет **Яндекс Директ** (расход за период) | Николай (или замещающий по регламенту) | Раз в отчётную неделю | В v1 **только Директ**; прочие площадки — позже |
| MKT-AD-MAP | Кабинеты **2ГИС** и **Яндекс.Карты** (реклама карточек) — **две** системы; в дашборд **одна сумма** вручную | Николай | Раз в отчётную неделю | Сложить расходы за ту же пн–вс |
| WEB-TRF-CH | **Яндекс.Метрика**: отчёт с метрикой **«Посетители»** и группировкой по источникам/каналам (см. §4); три строки формы | Николай | Раз в отчётную неделю | Строки: **Органика**, **CPC (Директ)**, **Прямые заходы** — см. §4.2 |
| WEB-BEH-BOUNCE, WEB-BEH-TIME | **Яндекс.Метрика**: показатели **по сайту в целом** за неделю (уровень **визита**); см. §4 | Николай | Раз в отчётную неделю | Длительность — **секунды**, одно поле (`WEB-BEH-TIME`) |
| REP-RATING, REP-REV-CNT | Экран **мобильного приложения** 2ГИС / Яндекс.Карт (карточка организации точки); кабинет не обязателен | Николай | **Понедельник**, снимок на **дату сохранения** | 4 сущности: 2 точки × 2 площадки; приросты — в системе |
| OZ-REV, OZ-ORD | Кабинет продавца **Ozon**: **операционная** выручка и заказы за календарную пн–вс (см. §4.3) | Николай | Раз в отчётную неделю | Источник правды по деньгам — **операционные** отчёты, не «финансовая» сверка v1 |
| OZ-RET-N, OZ-RET-SUM | Кабинет Ozon: возвраты за ту же неделю | Николай | Раз в отчётную неделю | Не уменьшают OZ-REV |
| OZ-AD-SPEND | Кабинет Ozon: расход на продвижение за период | Николай | Раз в отчётную неделю | Дублируется логически с §2.0 реестра; одно поле в блоке Ozon |

*Производные (`OFF-AVG-CHK`, `WEB-TRF-TOT`, `DER-*` и т.д.) в систему не вводятся — считает бэкенд по `metrics-registry.md`.*

## 4. Кабинеты и системы (as-is)

### 4.1 Сводная таблица

| Система | Назначение для дашборда | Что снимать / копировать | Примечание |
|---------|-------------------------|---------------------------|------------|
| Касса и Excel (точка) | OFF-REV, OFF-ORD | Итоги за пн–вс по регламенту точки | Детализация регламента — у владельца/ТПМ |
| Файл / учёт возвратов (точка) | OFF-RET-* | Количество и сумма за пн–вс | Отдельно от выручки |
| Яндекс Директ | MKT-AD-CTX | Расход за календарную неделю отчёта | Только v1-канал контекста |
| 2ГИС (рекламный кабинет) | Часть MKT-AD-MAP | Расход за неделю | Суммировать с Я.Картами |
| Яндекс.Карты (рекламный кабинет) | Часть MKT-AD-MAP | Расход за неделю | Суммировать с 2ГИС |
| Яндекс.Метрика | WEB-TRF-CH, WEB-BEH-* | См. §4.2 | Посетители по каналам ≠ визиты для bounce/time |
| Приложения 2ГИС и Яндекс.Карт | REP-* | Рейтинг и число отзывов с карточки | Снимок в пн |
| Ozon (личный кабинет продавца) | OZ-*, OZ-AD-SPEND | См. §4.3 | Названия разделов UI могут меняться — уточнять по факту |

### 4.2 Яндекс.Метрика: каналы и поведение

- **Период отчёта:** календарная **пн–вс** той же отчётной недели, что и в форме.
- **WEB-TRF-CH:** в отчёте «Источники / **По источникам трафика**» (или эквивалент с метрикой **Посетители**) взять **посетителей** за неделю и вписать в три строки:
  - **Органика** — трафик из поисковых систем (органический) по правилам Метрики для выбранного отчёта.
  - **CPC (Директ)** — платный контекст (в т.ч. переходы с Яндекс Директа), в границах того же отчёта.
  - **Прямые заходы** — прямой трафик по классификации Метрики.
- Если в интерфейсе Метрики группы называются иначе — **сохранить скрин/шпаргалку** для Николая; цель — стабильное правило «одна неделя — три числа».
- **WEB-BEH-BOUNCE** и **WEB-BEH-TIME:** стандартные отчёты по **визитам** за ту же неделю **по сайту** (например, обзор / поведение): **доля визитов с отказом** и **средняя длительность визита**; время переносить в форму **в секундах**.

### 4.3 Ozon (типовые места кабинета)

В продукте Ozon — **канал продаж** и **виртуальная точка** с отдельной выручкой и блоком ввода `OZ-*` (как у офлайн-точек по смыслу полей); см. `glossary.md`, `metrics-registry.md` §4.

Интерфейс Ozon меняется; ниже — **логика поиска**, не юридический текст UI.

- **OZ-ORD**, **OZ-REV (операционная):** разделы аналитики продаж / заказов (например, **Аналитика** → отчёты по **заказам** или **продажам**) за период **пн–вс**; брать показатели **заказов** и **операционной выручки** (как в отчёте «операционные» данные продавца), согласованные с еженедельным срезом.
- **OZ-RET-N**, **OZ-RET-SUM:** отчёт по **возвратам** за тот же календарный период.
- **OZ-AD-SPEND:** блок **рекламы / продвижения** (внутренняя реклама Ozon) — фактический **расход за неделю** по отчёту кабинета.

При расхождении отчётов приоритет для `OZ-REV` — **операционная** выручка (как согласовано с бизнесом).

## 6. Качество и пробелы

- **Скриншоты кабинетов** Ozon / Яндекс.Метрики на момент согласования PRD v0.5 **не** приложены; матрица §3 и §4 действует по текстовым правилам; уточнение пунктов меню в UI кабинетов — по мере появления материалов, **без** смены ID метрик.
- **Офлайн:** данные с точек сейчас собираются **не полностью**; продукт задаёт **поля**, полнота — организационный трек.
- **Возвраты офлайн:** отдельный файл; риск расхождений с кассой — не нормируется в v1.
- **Замещение маркетолога:** возможно; выбор **до трёх** прошлых недель в приложении снижает риск «пустой» недели.
- **Двойной учёт:** не смешивать расход Директа в `MKT-AD-CTX` и посетителей CPC в `WEB-TRF-CH` как одну сумму денег — это разные метрики.
- **Метрика:** сумма трёх строк «посетителей» может **не совпасть** с «всего посетителей» в другом отчёте из-за «других» источников; для `WEB-TRF-TOT` в продукте используется **сумма трёх введённых строк** (явное правило дашборда).
