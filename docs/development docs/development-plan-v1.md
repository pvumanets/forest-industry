## 3. Инфраструктура

| Параметр | Значение |
|----------|----------|
| **Репозиторий** | Личный GitHub (твой аккаунт). |
| **VPS** | **2 vCPU, 2 GB RAM, 60 GB SSD**; публичный IP **`66.151.34.138`**. |
| **TLS** | HTTPS на prod обязателен ([`dev-handoff-spec.md`](./dev-handoff-spec.md) NFR-06). Сертификат по IP (Let’s Encrypt ограничен) — варианты в [`runbook.md`](./runbook.md): самоподписанный с доверием на устройствах, коммерческий на IP, или позже **домен** + LE. |

### 3.1 Секреты и конфигурация (решение)

- **Секреты в git не коммитить:** пароли БД, `SECRET_KEY` для сессий, прод-пароли пользователей — только **переменные окружения** на сервере и локально в `.env` (файл в `.gitignore`).
- **Пример переменных** без значений — в **`.env.example`** в репозитории.
- Сиды с паролями для **первого выката** задаются через env или одноразовый скрипт; после стабилизации — **смена паролей** реальными (не из черновика плана).

> **Риск:** IP и сид-логины в markdown — для **закрытого** репо. При публикации репозитория — вычистить и ротировать пароли.

## 4. Технические решения (зафиксировано)

| Тема | Решение |
|------|---------|
| **Часовой пояс** | **`Asia/Yekaterinburg`** для границы «сегодня» и выбора **трёх предыдущих** отчётных недель; в коде и runbook явно. |
| **Политика паролей пользователей** | Минимум **10 символов**; **без обязательных** классов символов (буквы/цифры/спец — по желанию пользователя). Хранение только **Argon2id** или **bcrypt**. |
| **Ozon в модели данных** | **Виртуальная точка** в `outlets`: `code = OZON`, `is_virtual = true`; недельные метрики Ozon привязаны к этой записи (см. обновлённый [`architecture-draft.md`](./architecture-draft.md) §4.1). |
| **Изменение KPI в % на главной** | Отдельное правило (синхронизировано с [`dev-handoff-spec.md`](./dev-handoff-spec.md) §6.2): `previous > 0` — обычная формула %; `previous = 0` и `current = 0` → **«—»**; `previous = 0` и `current > 0` → подпись **«нов.»** / **«нов»** или **«—»** в бейдже, в **tooltip** или подписи: **«Рост с нуля»**. |
| **Макеты** | **Без Figma** в v1 — UI по ТЗ + shadcn/Tailwind. |
| **Подсказки у полей** | **Полные** тексты по [`data-map.md`](../product%20docs/data-map.md), стиль — [`ui-copy-guidelines.md`](../product%20docs/ui-copy-guidelines.md). |
| **Персональные данные** | Достаточно **внутренних договорённостей** закрытого контура; отдельная политика как у публичного SaaS не требуется для v1. |

## 5. Учётные записи сида (dev / первый выкат)

Роли продукта v1: `owner` | `marketer` | `site_manager`. Учётка **admin** в продуктовой матрице ролей не описана — для v1 трактуем как **`owner`** (тот же доступ, что у владельца), для удобства bootstrap и отладки.

| Логин | Роль приложения | Назначение |
|-------|-----------------|------------|
| `admin` | `owner` | Техническая / первая настройка |
| `evgeniy` | `owner` | Владелец бизнеса |
| `pavel` | `owner` | Владелец бизнеса |
| `marketing` | `marketer` | Маркетолог (Николай) |
| `manager` | `site_manager` | Управляющий точками — **обе** точки `NOVOGRAD` и `SVERDLOV` для проверки переключателя |

**Пароли для локальной разработки и первого деплоя** (задать через сид/env, **не** хардкод в репозитории при переходе на публичный git):

| Логин | Пароль (черновик) |
|-------|-------------------|
| admin | `admin_password` |
| evgeniy | `ev_password` |
| pavel | `pav_password` |
| marketing | `marketing_password` |

После выката на prod: **удалить или заменить** тестовые данные и **сменить пароли** на стойкие.

## 6. Тестовые данные (фикстуры)

- Нужны **синтетические данные на 6–8 недель** (офлайн по точкам, маркетинг, репутация, Ozon), чтобы проверить дашборды, графики и правила **%** и «рост с нуля».
- Реализация: **сид или фикстуры в репозитории** (команда в README). После первого реального использования на prod — **очистка** демо-данных по runbook.

Исторический импорт из Excel в v1 **нет** — короткая история на старте **подтверждена**.

## 7. Тестирование и качество

| Тема | Решение |
|------|---------|
| **Автотесты** | **Обязательны**; объём — **на усмотрение разработчика**, ориентир **«побольше»**: юнит + интеграционные API + ключевые сценарии фронта (по мере появления стека). |
| **Регрессия** (пояснение) | Повторный прогон тестов и короткий **ручной чеклист** после изменений, чтобы убедиться, что **старые сценарии не сломались**. Перед выкладкой на prod — полный прогон CI (когда появится) + смоук вручную. |
| **NFR** (LCP, p95 API из handoff) | Это **измеримые ориентиры** качества; **обязательная формальная замерка** перед каждым релизом не зафиксирована — по возможности проверять перед «большим» релизом (Lighthouse / логи). |

Критерии приёмки по фичам — [`dev-handoff-spec.md`](./dev-handoff-spec.md) §11 (A1–A10).
