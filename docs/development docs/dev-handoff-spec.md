## 1. Назначение документа

Перевод продуктовых требований в **конкретику для разработки и тестирования**: стек, экраны и их **состав** (§4.1), поля форм, API, правила валидации, визуальный ориентир (зародыш ERP), измеримые NFR и критерии приёмки. Документ — **источник правды для оценки сроков и спринтов**; при расхождении с PRD по смыслу приоритет у PRD, по технической детализации — у этого ТЗ после явного согласования.

## 2. Стек и репозиторий

Цель: **ощущение лёгкой ERP/операционной оболочки** — предсказуемый каркас (шапка, навигация, таблицы и формы в одном стиле), без визуального «стартапа ради стартапа».

| Параметр | Рекомендуемое значение | Комментарий |
|----------|------------------------|-------------|
| **Backend** | **Python 3.12**, **FastAPI**, **Pydantic v2**, **SQLAlchemy 2.x**, **Alembic** | REST/JSON; автогенерация OpenAPI из кода |
| **DB** | **PostgreSQL 15** | Миграции только через Alembic |
| **Frontend** | **React 18+**, **TypeScript**, **Vite**, **pnpm** | SPA |
| **Стили и UI** | **Tailwind CSS** v4 (тема на CSS variables / oklch) + **Radix UI** + **lucide-react** + **react-day-picker** | Источник правды по палитре и акцентам: [`ui-visual-guidelines.md`](../product%20docs/ui-visual-guidelines.md); минимализм ERP-оболочки |
| **Данные на клиенте** | **TanStack Query** | Кэш, повтор запросов, статусы загрузки |
| **Маршрутизация** | **React Router** v6+ | Защищённые маршруты по роли |
| **Графики** | **Recharts** или **Visx** (или **Tremor** при желании быстрее собрать дашборды) | Плавные линии/области, аккуратные оси, тултипы; на мобильном — упрощённые серии или скролл по оси X |
| **Infra** | **Docker Compose**, **nginx** (Alpine), TLS на prod | Локально → GitHub → prod VPS |
| **Репозиторий** | **GitHub** (URL организации/репо — по факту создания) | Монорепо `apps/web` + `apps/api` + `infra` — предпочтительно для v1 |
| **Порядок развёртывания v1** | **1)** разработка и проверка на **localhost** на машине ТПМ/разработчика **2)** код в **GitHub** **3)** prod на **VPS** по **публичному IP** (IP задаётся отдельно, в документ не вносится до получения) | На localhost HTTPS не обязателен; на prod — HTTPS по IP или по домену, когда появится |

**Альтернативы** (если команда сильнее в другом стеке): фронт **Vue 3 + Pinia + Vite**; бэкенд остаётся Python или **Node (NestJS)** — с сохранением контрактов API и схемы БД из этого ТЗ.

## 3. Роли и матрица доступа (детализация)

Роли в БД/API (enum): `owner` | `marketer` | `site_manager`.

| Роль | Экраны | API (принцип) |
|------|--------|----------------|
| **owner** | Логин, главный дашборд, все drill-down отчёты | Только **GET** дашбордов/отчётов; **нет** мутаций ввода |
| **marketer** | То же + единый поток **«Ввод за неделю»** (все блоки маркетолога) | GET дашбордов + GET/PUT своих пакетов ввода (маркетинг, сайт, репутация, Ozon) |
| **site_manager** | Логин, **ввод по офлайн-точке**, при необходимости краткий переход к дашборду если позже расширят — в v1 по PRD фокус на **ввод** | GET доступных недель и точек + GET/PUT `OFF-*` **только** для закреплённых `outlet_id` |

**Правило:** каждый запрос мутации проверяет роль и область (outlet) на сервере; клиентская скрытость кнопок — недостаточна.

## 4. Экраны и навигация

Общая оболочка после логина: **боковая навигация** (на мобильном — выдвижное меню), **верхняя панель** с названием продукта **Grove Pulse**, именем пользователя и выходом. Активный пункт подсвечен.

| # | Маршрут (черновик) | Роли | Описание |
|---|-------------------|------|----------|
| 1 | `/login` | все гости | Логин + пароль; редирект по роли после успеха |
| 2 | `/` | все авторизованные | Редирект: `owner` / `marketer` → `/dashboard`; `site_manager` → `/entry/offline` |
| 3 | `/dashboard` | owner, marketer | Главный дашборд владельца: период (неделя / месяц / квартал), блоки §6 |
| 4 | `/entry/week` | marketer | Выбор отчётной недели из **до трёх предыдущих** пн–вс, затем форма ввода (см. §5) |
| 5 | `/entry/offline` | site_manager | Переключатель точки (если 2), выбор недели, форма `OFF-*` |
| 6 | `/reports/site` | owner, marketer | Drill-down блока «Сайт»: графики, диапазон дат |
| 7 | `/reports/outlets` | owner, marketer | Drill-down «Точки»: по точкам, динамика `OFF-*` |
| 8 | `/reports/maps/2gis` | owner, marketer | Репутация 2ГИС: фильтр точки, оси по датам снимков |
| 9 | `/reports/maps/yandex` | owner, marketer | Репутация Яндекс.Карты |
| 10 | `/reports/ozon` | owner, marketer | Канал Ozon |
| 11 | `/reports/returns` | owner, marketer | Свод возвратов компании + разрез офлайн / Ozon |

Клик по карточке блока на `/dashboard` ведёт на соответствующий `/reports/...`.

### 4.1 Состав экранов (что видит пользователь)

Общее для всех экранов после входа: **сайдбар слева** (иконка/гамбургер на мобильном), **верхняя полоска**: слева логотип/название **Grove Pulse**, справа имя пользователя и **«Выйти»**. Контент — в основной области с отступами; на мобильном карточки **столбцом на всю ширину**.

**Пункты сайдбара по ролям**

| Роль | Пункты (сверху вниз) |
|------|----------------------|
| **owner** | Дашборд → Отчёты: Сайт, Точки, Карты 2ГИС, Карты Яндекс, Ozon, Возвраты |
| **marketer** | Дашборд → **Ввод за неделю** → те же Отчёты |
| **site_manager** | Только **Ввод по точке** (дашборды в v1 недоступны — старт с экрана ввода, [`prd.md`](../product%20docs/prd.md)) |

**`/login`** (гость)

- По центру экрана **одна карточка** (не на всю ширину на десктопе).
- Заголовок: «Вход в Grove Pulse».
- Поля: **Логин** (text), **Пароль** (password), кнопка **«Войти»**.
- Блок **ошибки** под формой (неверный логин/пароль, сеть) — текст на русском, без кода исключения.
- Нет ссылок «забыли пароль» (v1 — только через администратора).

---

**`/dashboard`** (`owner`, `marketer`)

- **Верх основной области:** переключатель **Неделя | Месяц | Квартал** + (желательно) селект **конкретного периода** из уже накопленной истории (например «9–15 марта 2026» / «Март 2026» / «Q1 2026»).
- Ниже — **сетка карточек-блоков** (на десктопе 1–2 колонки по ширине; порядок сверху вниз как в PRD):
  1. **Сайт** — заголовок блока; 3–4 KPI с подписями; для каждого: текущее значение крупно, строка «Было: …», бейдж **Δ%** или «—».
  2. **Точки** — свод по офлайн: **DER-REV-TOT** + строки по **NOVOGRAD** / **SVERDLOV** (выручка минимум); при необходимости заказы/средний чек; та же схема «было / %».
  3. **Карты — 2ГИС** — отдельная карточка: рейтинг/отзывы/прирост (по правилам агрегации); «было / %».
  4. **Карты — Яндекс** — отдельная карточка; аналогично.
  5. **Ozon** — выручка, заказы, реклама, средний чек; «было / %».
  6. **Возвраты** — суммы и штуки офлайн (все точки) + Ozon; «было / %».
- Вся карточка кликабельна (или явная ссылка «Подробнее») → соответствующий `/reports/...`.
- Пустое состояние: если истории мало — текст вроде «Недостаточно данных для сравнения» там, где нет прошлого периода.

---

**`/entry/week`** (`marketer`)

- **Шапка:** заголовок «Ввод за неделю».
- **Выбор недели:** сегмент или выпадающий список из **до трёх** доступных недель; подпись вида «Пн 03.03 — Вс 09.03».
- Одна **длинная форма** с якорными **секциями** (заголовки H2) и отступами:
  - «Реклама» — 2 поля (Директ, карты).
  - «Сайт: посетители» — 3 поля в ряд или столбцом на мобильном.
  - «Сайт: поведение» — отказы %, длительность сек.
  - «Репутация» — таблица **4 строки**: колонки «Точка», «Площадка», «Оценка», «Число отзывов»; строки = NOVOGRAD/SVERDLOV × 2ГИС/Яндекс. Над таблицей одно поле **«Дата снимка»** (date picker).
  - «Ozon» — 5 полей блоком.
- Внизу **закреплённая панель** (на мобильном): кнопка **«Сохранить»**; при невалидной форме — disabled или клик с прокруткой к первой ошибке.

**`/entry/offline`** (`site_manager`)

- Если у пользователя **2 точки** — **переключатель** (табы или segmented): «Новоградский» / «Свердловский» (подписи из справочника).
- Выбор **недели** — так же, как у маркетолога.
- Четыре поля ввода (`OFF-*`) + блок **«Считается автоматически»**: средний чек, средний чек возврата (если применимо).
- Кнопка **«Сохранить»**.

---

**`/reports/site`**

- Хлебные крошки: Дашборд → Сайт.
- Заголовок «Сайт: динамика».
- **Фильтры:** диапазон дат (от / до) или пресеты «последние 8 недель».
- **График:** линии или область — минимум `WEB-TRF-TOT` по времени; опционально второй ряд отказов/времени на второй оси или отдельная мини-серия.
- Под графиком — **таблица** по неделям (колонки: неделя, посетители всего, органика, CPC, прямые, отказы, время) при наличии места.

**`/reports/outlets`**

- Фильтр **«Точка: все | NOVOGRAD | SVERDLOV»**.
- Графики: минимум **выручка** по неделям; опционально второй график заказы/возвраты.
- Таблица по неделям и точкам.

**`/reports/maps/2gis`** и **`/reports/maps/yandex`**

- Фильтр **точка** (все / по одной).

---

**`/reports/ozon`**

- Диапазон дат; графики: выручка, заказы, реклама (как минимум выручка).

**`/reports/returns`**

- Сводная динамика: офлайн (сумма/шт по компании) и Ozon; можно одним составным графиком или двумя.

## 5. Формы ввода (недельный цикл)

### 5.1 Общие правила

- **Неделя:** в UI показывать диапазон **пн–вс** и ключ `week_start` = дата понедельника в **ISO `YYYY-MM-DD`** (календарная неделя по локали сервера — Россия).
- Доступные недели: **три последние завершённые** отчётные недели относительно «сегодня». **Часовой пояс:** **`Asia/Yekaterinburg`**; граница «сегодня» — начало календарного дня в этой зоне (зафиксировано в [`development-plan-v1.md`](./development-plan-v1.md); дублировать в runbook).
- Строка в `reporting_weeks` для выбранной доступной недели: при отсутствии создаётся **лениво** при первом GET/PUT ввода по этой неделе (идемпотентно, без дубля по `week_start_date`).
- Все перечисленные поля **обязательны**; **0** допустим для числовых (включая деньги).
- Ошибка сохранения: сообщение вроде «Заполните все поля» + подсветка пустых; сервер **422** с телом валидации по полям.
- Под каждым полем или блоком — **полный текст подсказки** из [`data-map.md`](../product%20docs/data-map.md) (стиль см. [`ui-copy-guidelines.md`](../product%20docs/ui-copy-guidelines.md)).
- **Идемпотентность:** повторный PUT с тем же `week_start` и тем же разрезом **перезаписывает** сохранённый пакет (одна версия на пару ключей).

### 5.2 Руководитель точки (`site_manager`) — форма офлайн

Контекст: выбранная **точка** (`NOVOGRAD` | `SVERDLOV`) и **неделя**.

| Поле (ID) | Подпись UI | Тип | Валидация |
|-----------|------------|-----|-----------|
| OFF-REV | Выручка за неделю | decimal ≥ 0, 2 знака | обязательно |
| OFF-ORD | Заказы, шт. | целое ≥ 0 | обязательно |
| OFF-RET-N | Возвраты, количество | целое ≥ 0 | обязательно |
| OFF-RET-SUM | Возвраты, сумма | decimal ≥ 0 | обязательно |

Производные показывать **read-only** под формой после сохранения или live: `OFF-AVG-CHK`, `OFF-RET-AVG` (если знаменатель > 0).

### 5.3 Маркетолог — единая форма «Неделя»

**Блок «Реклама»**

| ID | Подпись | Тип |
|----|---------|-----|
| MKT-AD-CTX | Расход, Яндекс Директ (неделя) | decimal ≥ 0 |
| MKT-AD-MAP | Расход, реклама на картах 2ГИС + Я.Карты (сумма за неделю) | decimal ≥ 0 |

**Блок «Сайт — трафик»** (три строки, подписи фиксированы)

| ID | Подпись | Тип |
|----|---------|-----|
| WEB-TRF-CH | Посетители, органика | целое ≥ 0 |
| WEB-TRF-CH | Посетители, CPC (Директ) | целое ≥ 0 |
| WEB-TRF-CH | Посетители, прямые заходы | целое ≥ 0 |

В API — массив из трёх объектов `{ channel: enum, visitors }` или три именованных поля; стабильные ключи: `organic`, `cpc_direct`, `direct`.

**Блок «Сайт — поведение»**

| ID | Подпись | Тип |
|----|---------|-----|
| WEB-BEH-BOUNCE | Отказы, % (по сайту) | decimal 0–100 |
| WEB-BEH-TIME | Средняя длительность визита, сек | decimal ≥ 0 |

**Блок «Репутация»** — сетка **2 точки × 2 площадки** (2ГИС, Яндекс.Карты). На ячейку:

| ID | Подпись | Тип |
|----|---------|-----|
| REP-RATING | Средняя оценка (на дату снимка) | decimal |
| REP-REV-CNT | Количество отзывов | целое ≥ 0 |

**Дата снимка:** одна дата на всю форму репутации за проход (обычно дата ввода в пн) — поле `snapshot_date` (date); либо дата на каждую ячейку, если решат гибче — v1 достаточно **одной даты на сохранение** для всех четырёх ячеек.

**Блок «Ozon»**

| ID | Подпись | Тип |
|----|---------|-----|
| OZ-REV | Выручка Ozon (операционная) | decimal ≥ 0 |
| OZ-ORD | Заказы, шт. | целое ≥ 0 |
| OZ-RET-N | Возвраты, количество | целое ≥ 0 |
| OZ-RET-SUM | Возвраты, сумма | decimal ≥ 0 |
| OZ-AD-SPEND | Реклама на Ozon | decimal ≥ 0 |

Кнопка **«Сохранить неделю»** активна только если все поля маркетолога для выбранной недели заполнены (после нормализации пустых строк в ошибку, не в 0).

## 6. Дашборды и виджеты (MVP)

### 6.1 Визуальный язык (зародыш ERP + современные графики)

- **Фон:** нейтральный светлый по токенам `--background` / `--muted`; карточки — `--card` с **рамкой** `border-border`; тени точечно, без «плавающих» панелей на каждом виджете.
- **Акцент:** фирменный **#FACE08** как маркетинговый ориентир; в UI основной акцент — токен **`primary`** (OKLCH в [`apps/web/src/index.css`](../../apps/web/src/index.css)). Подробности — [`ui-visual-guidelines.md`](../product%20docs/ui-visual-guidelines.md). На больших заливках primary текст **тёмный** (контраст).
- **Типографика:** **Montserrat** (UI), при необходимости **Merriweather** / **JetBrains Mono** — см. `ui-visual-guidelines`; цифры **табличные** (`tabular-nums`).
- **Карточки KPI:** крупное текущее значение; под ним строка «прошлый период: X» и **бейдж %** (↑/↓ цветом нейтральным: рост не всегда «хорошо»); если % невозможен — **«—»**.
- **Графики:** сетка минимальная; легенда компактная; анимация появления лёгкая; на мобильном — меньше подписей осей, сохранить читаемость ключевых значений.

### 6.2 Главный дашборд (`/dashboard`)

Переключатель периода: **Неделя | Месяц | Квартал**. Якорь периода — календарный (текущая завершённая неделя / текущий месяц / квартал или выбор из селекта — реализация: **выбор конкретной недели/месяца/квартала** из доступной истории).

**Ключевые KPI по блокам** (все с **предыдущим периодом** и **%** где осмысленно — [`prd.md`](../product%20docs/prd.md) §5.3):

| Блок | KPI на карточке (пример набора 3–4) |
|------|-------------------------------------|
| **Сайт** | `WEB-TRF-TOT`; `WEB-BEH-BOUNCE`; `WEB-BEH-TIME` |
| **Точки** | `DER-REV-TOT`; по подстрокам: выручка **NOVOGRAD**, **SVERDLOV**; при желании `DER-ORD-TOT` |
| **Карты → 2ГИС** | Среднее `REP-RATING` по точкам (или две строки); суммарный `REP-REV-CNT`; суммарный прирост отзывов за период |
| **Карты → Яндекс** | Аналогично для Яндекс.Карт |
| **Ozon** | `OZ-REV`, `OZ-ORD`, `OZ-AD-SPEND`, `OZ-AVG-CHK` |
| **Возвраты** | Σ `OFF-RET-SUM` по точкам + `OZ-RET-SUM`; Σ `OFF-RET-N` + `OZ-RET-N` |

**Процент изменения и особые случаи** (единый стиль по всему приложению; подробнее [`ui-copy-guidelines.md`](../product%20docs/ui-copy-guidelines.md) §4):

| Условие | Бейдж / подпись | Подсказка (tooltip или мелкий текст) |
|---------|-----------------|--------------------------------------|
| `previous != 0` | `((current - previous) / abs(previous)) * 100` | По желанию UI |
| `previous == 0` и `current == 0` | **—** | — |
| `previous == 0` и `current > 0` | **«нов.»** / **«нов»** или **«—»** (один вариант на всё приложение) | **«Рост с нуля»** |

**Репутация в месяце/квартале:** агрегат «последний снимок в периоде» по каждой паре точка×площадка, затем сводка для карточки ([`metrics-registry.md`](../product%20docs/metrics-registry.md) §3).

### 6.3 Drill-down

- Фильтры: диапазон дат (от-до по неделям или календарю), для отчётов точек — выбор **точки** или «все».
- Минимум **один график** временных рядов на экран; таблица детализации под графиком — по желанию, желательно для ERP-ощущения.

## 8. API-спецификация

Префикс **`/api`**. Формат JSON, кодировка UTF-8.

**Auth**

| Метод | Путь | Описание |
|-------|------|----------|
| POST | `/api/auth/login` | body: `{ "login", "password" }` → **Set-Cookie**: opaque session id (см. §9); в body **не** отдавать токены |
| POST | `/api/auth/logout` | Удаление строки сессии в БД + очистка cookie |
| GET | `/api/auth/me` | Текущий пользователь, роль, outlets[] для site_manager |

**Справочники и недели**

| Метод | Путь | Описание |
|-------|------|----------|
| GET | `/api/weeks/selectable` | Список до 3 отчётных недель `{ week_start, label }` |
| GET | `/api/outlets` | Список точек (физические; виртуальный Ozon не обязателен в списке для ввода офлайн) |

**Ввод**

| Метод | Путь | Описание |
|-------|------|----------|
| GET/PUT | `/api/submissions/offline/{week_start}/{outlet_code}` | Пакет `OFF-*` |
| GET/PUT | `/api/submissions/marketing/{week_start}` | Маркетинг + сайт + репутация + Ozon (одно тело JSON по секциям) |

**Дашборды и отчёты**

| Метод | Путь | Описание |
|-------|------|----------|
| GET | `/api/dashboard/summary` | query: `period=week|month|quarter`, `anchor=...` → блоки и KPI + previous + pct |
| GET | `/api/reports/{topic}/series` | query: `from`, `to`, фильтры — серии для графиков (`topic` = site \| outlets \| maps-2gis \| maps-yandex \| ozon \| returns) |

Коды ответов: **401** не авторизован, **403** запрещено, **404** нет данных / неделя вне доступа, **422** валидация.

OpenAPI: публиковать на `/api/openapi.json` (FastAPI по умолчанию).

## 9. Авторизация

- **Решение v1:** **сессии в PostgreSQL** (`user_sessions`), **без JWT** в качестве основного механизма.
- **Cookie:** имя, например `gp_session`; значение — **криптостойкая случайная строка** (≥ 32 байта энтропии, напр. base64url). В БД хранить **`token_hash` = SHA-256(значение cookie)** и по нему искать сессию (сырой токен в БД не хранить).
- **Атрибуты cookie:** **HttpOnly**, **SameSite=Lax**; **Secure** — **вкл на prod** (HTTPS); на **localhost** допускается без Secure для разработки.
- **Срок жизни сессии:** **14 суток** с последней активности (**sliding**: обновлять `last_seen_at` на каждый успешный запрос с валидной сессией); по истечении — 401, редирект на логин.
- **Logout:** удалить строку из `user_sessions`, cookie очистить.
- **Пароли:** хэш **Argon2id** или **bcrypt**; в открытом виде не логировать.
- **Политика паролей v1:** минимум **10 символов**; **обязательных** классов символов (буквы/цифры/спецсимволы) **нет** — пользователь может задать любую строку длины ≥ 10 ([`development-plan-v1.md`](./development-plan-v1.md)).
- **Сброс пароля:** вручную через сид/скрипт (Павел Уманец) — без self-service UI.
- **CORS:** на localhost — `http://localhost:<порт>` фронта; на prod — origin по фактическому URL (IP или домен).

## 13. Операционка и доступы

- **Учётные записи пользователей** создаёт и раздаёт **Павел Уманец** (согласование логинов и ролей с владельцем продукта); техническая установка пароля — через сид/скрипт развёртывания по запросу.
- **Развёртывание:** сначала **localhost** → затем код в **GitHub** (личный репозиторий владельца) → **prod на VPS**. Публичный IP и TLS — см. [`runbook.md`](./runbook.md) и [`development-plan-v1.md`](./development-plan-v1.md).
- **Резервное копирование БД:** полный **`pg_dump` раз в неделю**, **по воскресеньям в 18:00 GMT** (UTC); cron на сервере: `0 18 * * 0` при интерпретации cron в **UTC**. Хранение и ретенция — в runbook ([`architecture-draft.md`](./architecture-draft.md) §9).
