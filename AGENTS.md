# Инструкции для AI-агента (Cursor) — Grove Pulse

Документ задаёт **обязательное** поведение агента при работе с этим репозиторием. Детали продуктовых задач и таблица артефактов — в [`docs/product docs/agent_product.md`](docs/product%20docs/agent_product.md).

---

## 1. Роль и приоритеты

1. **Источник правды по продукту и данным** — [`docs/product docs/`](docs/product%20docs/) (PRD, MVP, реестр метрик, data map, глоссарий и т.д.).
2. **Источник правды по реализации** — [`dev-handoff-spec.md`](docs/development%20docs/dev-handoff-spec.md), [`architecture-draft.md`](docs/development%20docs/architecture-draft.md), [`development-plan-v1.md`](docs/development%20docs/development-plan-v1.md), [**пофазный план**](docs/development%20docs/development-phases-v1.md).
3. При конфликте смысла с PRD — приоритет у PRD; по технической детализации после явного согласования — у handoff-спека.
4. Язык ответов пользователю: **русский**, если пользователь пишет по-русски.
5. **Несколько версий Python на машине:** для этого репозитория целевая версия — **3.12** (как в CI и `requires-python` в `apps/api/pyproject.toml`). Если установлены и 3.12, и 3.14 (и др.), создавайте venv явно от 3.12 (например `py -3.12 -m venv .venv` в Windows) и не делайте вывод о версии только по одному вызову `python`.
6. Меняй только то, что нужно для задачи; не делай сопутствующих «улучшений» и рефакторингов без запроса.

## 3. Постоянное тестирование и проверки после каждой итерации

**Правило:** после **каждой завершённой итерации** (законченный логический шаг: фича, исправление, рефакторинг по задаче) агент **обязан** выполнить доступные автоматические проверки и **сообщить пользователю результат** (успех / ошибки / что не запускалось и почему).

**Итерация** здесь — не «каждый отдельный файл», а **согласованный кусок работы**: например, реализация эндпоинта + тесты к нему, или правка формы + прогон фронтовых тестов.

### 3.1 Что запускать

1. **Если в репозитории уже есть скрипты проверки** (корневой или пакетный `package.json`, `pyproject.toml`, `Makefile`, CI-конфиг) — используй **их**: тесты, линтеры, проверка типов, сборка. Предпочитай команды из README корня или из `apps/web` / `apps/api`, если монорепо уже развёрнут.
2. **Целевой стек проекта** (из handoff): фронт **pnpm** + **Vite** + **TypeScript**; бэкенд **Python 3.12** + **FastAPI**. Когда появятся пакеты, ориентир команд:
   - API: `pytest` (или эквивалент из `pyproject.toml`), при наличии — `ruff check`, `mypy`;
   - Web: `pnpm lint`, `pnpm test` / `pnpm exec vitest`, при наличии — `pnpm build`;
   - при корневом оркестраторе — одна команда вроде `pnpm test` / `make test` из документации репозитория.
3. **Монорепо (после фазы 1):** из `apps/api` — `python -m pytest`, `python -m ruff check src tests` (после `pip install -e ".[dev]"`); из `apps/web` — `pnpm test`, `pnpm build` (после `pnpm install`). Корневого единого скрипта нет; см. раздел «Проверки» в корневом `README.md`. CI: `.github/workflows/ci.yml`.
4. **Пока кода нет** — явно укажи в ответе, что автопроверки не настроены; предложи добавить минимальный набор (lint + тесты) при появлении первого кода.
5. **Если команда падает** — не скрывай вывод; разбери причину, исправь или объясни блокер. Не заявляй задачу выполненной на основании только «код написан».

### 3.2 Когда можно не гонять полный набор

- Чистые правки **только** markdown в `docs/` без изменения кода — достаточно явно написать, что автотесты не применимы.
- Задача явно помечена как черновик без завершения — всё равно по возможности прогони затронутые проверки для изменённых пакетов.

### 3.3 Поддержка этого раздела

Когда в репозитории появляется **единая команда** для всех проверок (например `pnpm verify` в корне или секция в README), **добавь одну строку** в §3.1 этого файла или в корневой README со точной командой, чтобы следующие итерации не гадали.

## 4. Краткий чеклист перед сдачей задачи

- [ ] Требования сверены с `docs/` (product + development по необходимости).
- [ ] Запущены релевантные тесты/линт/сборка; результат сообщён пользователю.
- [ ] Нет лишних файлов и отладочного мусора; секреты не закоммичены.
- [ ] При изменении контрактов API или схемы БД — обновлены документы или OpenAPI/миграции согласованы с `dev-handoff-spec.md`.
- [ ] При **завершении реализации очередной фазы** по [`development-phases-v1.md`](docs/development%20docs/development-phases-v1.md) — обновить **три места** в том файле: строка фазы в **сводной таблице** в начале, **`Статус реализации`** в § фазы, **чеклист приёмки** (`[x]`). Не оставлять только правку в середине документа без таблицы.
